## 1. Codex 對 §8 七個問題的裁決摘要

### ① Consensus 去 Java 化

**裁決**：要。

**原因**：Java 結構分目前在 `ConsensusScoringEngine` 被算兩次：
- 當 `base = min(java, claude, codex)` 的下限
- 當 `|java-claude|` / `|java-codex|` 的分歧懲罰來源

**對強股特別傷**：AI 雙高（claude=9, codex=8）但 java 結構分中等（5.2）的標的，會被雙重打壓。

**最小改法**：
```
consensus_score = min(claude, codex) - |claude - codex| × weight
```
Java 結構分保留在 `WeightedScoringEngine` 或獨立 `java_structure_score`，不再進共識懲罰。

### ② Top 10 問題的本質

**裁決**：流程太嚴是主因，`entryTriggered` 等欄位有 bug 是次因。

**具體處置**：
- `entryTriggered` 移到 `ExecutionTimingEngine`（非 FinalDecision hard gate）
- `mainStream` 從 hard block 改 **penalty 或 ranking boost**
- `belowOpen` / `belowPrevClose` 在 09:00 前後不能當最終否決

### ③ Session-Aware 拆分

**裁決**：必拆，不是優化。

- **PREMARKET（08:30-09:00）**：只能做 bias / report，不可出 ENTER/REST
- **OPEN_VALIDATION（09:00-09:30）**：更新 open 後資料，可回 WAIT
- **LIVE_TRADING（09:30+）**：才允許正式 ENTER / REST

### ④ `WAIT` 決策 type

**裁決**：新增 `WAIT`，不偷用 REST。

**語義區分**：
- `REST` = 今天不做（已決策）
- `WAIT` = 還沒到可決策時間（未決策）
- UI / notification / scheduler 都應區分
- **`WAIT` 不算 bearish，不應進「今日休息」統計**

### ⑤ Penalty 下修時機

**裁決**：現在就降（Phase A，不等 Phase B）。

**原因**：架構修對了但 penalty 維持原強度，系統還是會把 A/B 候選壓成 C。

**順序**：
1. Session-Aware
2. Consensus 重構
3. Penalty 下修

三者都屬 Phase A。

### ⑥ 實作順序

**裁決**：Session-Aware 放 Phase A #1 正確。

**原因**：Session-Aware 不先做，後續改 consensus / penalty / regime 都在髒輸入上工作（試撮資料污染）。

### ⑦ 全部實作分工

**直接丟 Claude 開工（Phase A）：**
1. Session-Aware 全套（enum / workflow / jobs / WAIT decision type）
2. ConsensusScoringEngine 重構（Java 移出共識懲罰）
3. Penalty 下修（NOT_IN_FINAL_PLAN / RR_BELOW_MIN / ENTRY_TOO_EXTENDED）

**第二批（Phase B）：**
4. Regime 下游消費
5. Decision Lock 自動清除
6. Portfolio 強替換

**Austin 需先定：**
- `WAIT` 的使用者體驗（LINE / Dashboard / 統計）
- OPEN_VALIDATION 終點切點

## 3. v2.7 Phase A 實作細節

### 3.1 新增 `MarketSession` enum

**路徑**：`src/main/java/com/austin/trading/domain/enums/MarketSession.java`

```java
package com.austin.trading.domain.enums;

import java.time.LocalTime;

public enum MarketSession {
    PREMARKET,        // 08:30-09:00，僅允許 bias + 候選清單
    OPEN_VALIDATION,  // 09:00-09:30，允許真實資料 scoring
    LIVE_TRADING;     // 09:30 之後，允許完整決策

    public static MarketSession fromTime(LocalTime now) {
        if (now == null) return LIVE_TRADING; // 保守 fallback
        if (now.isBefore(LocalTime.of(9, 0)))  return PREMARKET;
        if (now.isBefore(LocalTime.of(9, 30))) return OPEN_VALIDATION;
        return LIVE_TRADING;
    }

    public boolean allowsFinalDecision() {
        return this == LIVE_TRADING;
    }

    public boolean allowsScoringUpdate() {
        return this != PREMARKET;
    }
}
```

### 3.2 擴充 `FinalDecisionResponse` 加 `WAIT`

**路徑**：`src/main/java/com/austin/trading/dto/response/FinalDecisionResponse.java`

`decision` 欄位新增第四種值：
- `ENTER` — 可進場
- `REST` — 今日不做（已決策）
- `WAIT` — **新增**：未到決策時間
- （既有 decision 字串保留向下相容）

**規則**：
- `WAIT` 時 `selectedStocks` 永遠空
- `WAIT` 時 `rejectedReasons` 可包含 `["session=PREMARKET"]` 或 `["session=OPEN_VALIDATION"]`
- `summary` 給 UI 顯示用：`盤前觀察中` / `開盤驗證中，等 09:30 決策`

### 3.3 `FinalDecisionEngine` session 檢查

**路徑**：`src/main/java/com/austin/trading/engine/FinalDecisionEngine.java`

在 `evaluate(FinalDecisionEvaluateRequest request)` 開頭加：

```java
// v2.7: Session-Aware gate
MarketSession session = MarketSession.fromTime(LocalTime.now());
if (!session.allowsFinalDecision()) {
    log.info("[FinalDecisionEngine] WAIT: session={}", session);
    return new FinalDecisionResponse(
            "WAIT",
            List.of(),
            List.of("session=" + session),
            session == MarketSession.PREMARKET
                ? "盤前觀察中，09:30 再出決策。"
                : "開盤驗證中，等 09:30 決策。"
    );
}
```

### 3.4 `ConsensusScoringEngine` 去 Java 化

**路徑**：`src/main/java/com/austin/trading/engine/ConsensusScoringEngine.java`

**舊算法**：
```
base = min(java, claude, codex)
penalty = |java-claude|×pjc + |java-codex|×pjx + |claude-codex|×pcx
consensus = base - penalty
```

**新算法**：
```
if (claude 或 codex 有 null) {
    // 沒有 AI 共識可言，回傳 null（呼叫方 fallback 到 weighted score）
    return null;
}
base = min(claude, codex)
penalty = |claude - codex| × consensus.penalty_cx
consensus = max(base - penalty, 0)
```

**config 調整**：
- `consensus.penalty_jc`（Java-Claude）**廢棄**（保留 DB 記錄但不讀）
- `consensus.penalty_jx`（Java-Codex）**廢棄**
- `consensus.penalty_cx`（Claude-Codex）維持 0.20

### 3.5 Penalty 下修

**Live DB PUT + V22 migration**：

| config | 現值 | 新值 | 理由 |
|--------|-----:|-----:|------|
| `penalty.not_in_final_plan` | 0.5 | **0.1** | candidate 建立時 final_plan 常 false，不應常態扣分 |
| `penalty.rr_below_min` | 1.5 | **0.5** | RR 用靜態 entry 算不準，降 penalty 待 Phase B 動態 RR |
| `penalty.entry_too_extended` | 1.0 | **0.0** | 移到 `ExecutionTimingEngine` 處理，不在 VetoEngine 扣分 |

### 3.6 `entryTriggered` 移到 `ExecutionTimingEngine`

**拆分**：
- `FinalDecisionEngine.validateBasicConditions` **移除** `entryTriggered` 檢查
- `ExecutionTimingEngine.evaluate` 已有 entry 判斷邏輯（breakout/pullback trigger），**強化並由它擋 entry 訊號未確認的 candidate**
- `decision.require_entry_trigger` config 保留但語義改為「ExecutionTimingEngine 是否強制 trigger」

### 3.7 `mainStream` 從 hard block 改 ranking boost

**修改**：
- `FinalDecisionEngine.validateBasicConditions` **移除** `!mainStream → reject`
- 改在 scoring 階段：`mainStream=true` → `finalRank += ranking.main_stream_boost`（預設 0.3）
- 新 config：`ranking.main_stream_boost = 0.3`

### 3.8 `belowOpen` / `belowPrevClose` session 限制

**修改**：
- session=PREMARKET：**不計算** `belowOpen` / `belowPrevClose`（試撮無意義）
- session=OPEN_VALIDATION：計算但**只用於警示、不作為否決**
- session=LIVE_TRADING：維持現行 hard block

### 3.9 LINE 通知 / Dashboard 改動

**LINE**：
- 08:30 PremarketNotifyJob 發「盤前 bias」訊息，含 market_bias + 預期主流族群 + 候選清單
- 09:30 FinalDecision0930Job 發「最終決策」訊息（原行為不變）
- **刪除**「WAIT」相關通知（不發 WAIT 單獨訊息）

**Dashboard**（`src/main/resources/static/index.html`）：
- 新增 `PREMARKET BIAS` 頁面區塊（藍底）
- 08:30-09:30 時段顯示 BIAS + 09:30 倒數
- 09:30 後切換回原 final decision 頁面

**統計**：
- `DailyPnlService` / 「今日休息」計算：`decision=WAIT` 不計入休息
- 統計面板顯示 `WAIT` 獨立一類（非空手也非 REST）

### 3.10 Orchestration step 調整

**修改**：
- `OrchestrationStep.FINAL_DECISION` 的 `markDone` 條件：**09:30 之後才接受 DONE**
- 09:30 前 FinalDecisionEngine 回 WAIT 時，orchestration step 維持 `WAITING` 狀態
- 新增狀態 `WAITING`（區隔 `PENDING` 尚未觸發、`RUNNING` 執行中、`WAITING` 等時機）
