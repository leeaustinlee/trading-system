# v2.2 UI/UX 調整規格

> 目標：修正 v2.1 後 Dashboard / 持倉 / 損益 / 決策歷史 / AI 研究 共 5 個頁面的資訊顯示與操作缺陷。
> 所有修改僅影響前端呈現與少量 API/後端輔助邏輯；不改動 v2.1 已完成的 AI orchestration 流程。

---

## A. 總覽（Dashboard）

### A-1. 市場狀態：顯示邏輯改為「今日優先，今日缺則退回最新一筆」

**問題**：目前 `GET /api/market/current` 可能回傳最新一筆 `market_snapshot`（不限日期），非交易日或早盤前會看到昨日殘留資料。

**規格**：
- 後端：`MarketDataService.getCurrentMarket()` 新增參數或新 endpoint `GET /api/market/today`：
  1. 先查 `trading_date = today` 的最新一筆（以 `created_at DESC`）。
  2. 若無，退回「最近一個交易日」的最新一筆，response header/payload 加 `stale: true` + `stale_reason: NO_TODAY_DATA`。
- 前端：`d-marketStale` 原本的提示 pill 顯示邏輯：
  - `stale=false` → 隱藏。
  - `stale=true` → 顯示灰色 pill「最新資料：YYYY-MM-DD（非今日）」。
- Dashboard 頁載入改呼叫 `/api/market/today`（或沿用 `/current` 加上 `preferToday=true` 查詢參數）。

**驗收**：
- 假日開系統 → 顯示最近交易日資料 + stale pill。
- 交易日 08:10 前 → 若已有 PREMARKET snapshot 則顯示；否則退回昨日 + stale pill。
- 交易日盤中 → 顯示當日且 stale=false。

---

### A-2. 近期通知：顯示 5 筆、類型改顯示中文

**問題**：
1. 目前 `d-notificationsBody` 沒限制筆數。
2. `notificationType` 顯示為 `REVIEW_1400`、`PREMARKET_0830` 等 enum，未透過 `ZH_MAP` 翻譯。

**規格**：
- 後端：`GET /api/notifications?limit=5&order=desc` 已支援（或新增 limit 參數）。
- 前端 `ZH_MAP` 擴充通知類型：

  | 原碼 | 顯示 |
  |---|---|
  | `PREMARKET_0830` | 盤前分析 |
  | `FINAL_DECISION_0930` | 09:30 最終決策 |
  | `HOURLY_GATE` | 整點閘道 |
  | `MONITOR_5M` | 5 分鐘監控 |
  | `REVIEW_1400` | 14:00 交易檢討 |
  | `MIDDAY_1100` | 盤中分析 |
  | `TOMORROW_PLAN_1800` | 明日策略 |
  | `POSTMARKET_1530` | 盤後選股 |
  | `SYSTEM_ALERT` | 系統警示 |
  | `POSITION_ALERT` | 持倉警報 |

- 前端：呼叫時帶 `limit=5`，另加 `zh-通知類型-映射` 區塊到 `ZH_MAP`。
- 決策歷史頁（A-4）的「近期通知」也共用此映射。

**驗收**：Dashboard 近期通知固定 5 筆、類型欄顯示「盤前分析」「14:00 交易檢討」等中文。

## B. 持倉

### B-1. 出清支援分段（部分張數）

**問題**：目前「出清」UI 只能一次關倉全部，分段停利無法在系統內記錄。

**規格**：
- API：新增 `POST /api/positions/{id}/partial-close`
  ```json
  { "qty": 2, "price": 25.70, "reason": "TAKE_PROFIT_1" }
  ```
  - 後端：若 `qty < position.qty` → 建立出場 trade log + 更新 `position.qty -= qty`、`position.realizedPnl += (price - avgCost) * qty`。
  - 若 `qty >= position.qty` → 視同全部出清，維持原有 close 行為。
- 前端：
  - 原「出清」按鈕改為「出場」下拉：`部分出清` / `全部出清`。
  - 「部分出清」開 modal，輸入張數 + 出場價 + 原因（下拉：停利一/停利二/停損/移動停利/手動）。
- 後端保留原 `POST /api/positions/{id}/close`（全部出清）不變。

**驗收**：
- 3320 股分段：先 1000 股停利、再 1000 股、最後 1320 股移動停利 → 3 筆 trade log、position.qty 從 3320 → 2320 → 1320 → 0。

### B-2. 停損停利 AI 每日分析調整

**問題**：使用者未設定 stop_loss / take_profit 時，畫面顯示「–」，沒有 AI 建議值。

**規格**：
- 新欄位於 `position`：`suggested_stop_loss`, `suggested_take_profit_1`, `suggested_take_profit_2`, `suggested_updated_at`, `suggested_source`（AI / RULE）。
- 新 Job `PositionSuggestionJob`（每日 14:30 MON-FRI）：
  1. 讀所有 OPEN 持倉。
  2. 若 `stop_loss` 為 null → 呼叫 `PositionSuggestionService.compute(symbol, avgCost, currentPrice, direction)`：
     - RULE 策略：中短線 avgCost * 0.94 / 波段 avgCost * 0.93（依 holdingDays 判斷）。
     - 可選 AI 策略（Claude 研究）：從 `claude-research-latest.md` 讀該檔建議價。
  3. 寫回 `suggested_*` 欄位。
- 前端「持倉」表格：
  - 停損欄：若 `stop_loss` 有值 → 直接顯示；若無 → 顯示 `suggested_stop_loss`（灰色斜體 + 標籤「AI 建議」）。
  - 停利同上。
  - 點建議值 → 彈出「確認採用此建議」按鈕 → 寫入正式欄位。

**驗收**：
- 新買 00631L 未設停損 → 14:30 後顯示 AI 建議 -6%。
- 使用者按「採用建議」→ 寫入 stop_loss，持倉監控 Job 會用此值。

## E. AI 研究

### E-1. 「觸發 AI 研究」功能說明與使用指引

**問題**：目前按鈕會寫入 `claude-research-request.json`，但使用者不知道是否真會被 Claude 讀取、何時讀取。

**現況**（確認過）：
- 按鈕呼叫 `POST /api/ai/trigger/{premarket|stock|final-decision}` → `ClaudeCodeRequestWriterService.writeRequest()` 寫入 `D:\ai\stock\claude-research-request.json`。
- 但 **實際 Claude Code 排程 Agent 目前並未訂閱此檔觸發**（排程 Agent 固定時間跑，不 watch 檔案）。
- 檔案只是「備忘錄」，Claude Code 開啟時才會讀。

**規格**：
- 保留現有按鈕，但每個按鈕後加 `?` 圖示 → hover/click 顯示說明 popup：

  ```
  此按鈕會寫入 D:\ai\stock\claude-research-request.json，
  內容為你想請 Claude 研究的題目。

  使用方式：
  1. 按此按鈕 → 檔案更新
  2. 到 Claude Code 視窗，輸入「讀 claude-research-request.json 並執行研究」
  3. Claude 完成後會寫回 claude-research-latest.md
  4. 回到本頁「匯入研究報告」按鈕 → 寫入 DB

  注意：目前 Claude Code 排程 Agent 依固定時間執行，
  此按鈕只影響「手動觸發」流程，不會改變排程。
  ```

- 後續若要做「檔案變動自動觸發」需另外規劃（Claude Code hook / Task queue）— 不在 v2.2 範圍。

**驗收**：每個觸發按鈕右側有 `?` icon，點擊彈出說明。

---

### E-2. AI 研究紀錄：顯示重點階段

**問題**：目前 `#ai-logsBody` 可能列出所有 request / log，欄位雜亂。

**規格**：
- 後端：`GET /api/ai/logs?date=&type=` 已支援；新增回傳欄位：
  - `stage`: `REQUEST_WRITTEN` / `CLAUDE_RUNNING` / `CLAUDE_DONE` / `CODEX_RUNNING` / `CODEX_DONE` / `FINALIZED` / `IMPORTED`。
  - `durationMs`: 階段耗時（若可推算）。
  - `summary`: 該階段關鍵訊息（入選股、reject 原因）。
- 前端：以時間軸 timeline 顯示（每一筆 task 一組卡片，卡片內垂直列出階段 pill + 時間 + summary）：

  ```
  [POSTMARKET | 2026-04-20]
    ✅ 請求寫入     15:05:01  20 檔候選寫入
    ✅ Claude 研究  15:20:12  12 檔 PASS / 8 檔 REJECT
    ✅ Codex 最終   15:28:44  最終 3 檔：2337, 2383, 3017
    ✅ FINALIZED    15:30:02  已通知 LINE
  ```

- 不顯示中間 raw JSON；點「展開」才看原始內容。

**驗收**：AI 研究紀錄頁改成 timeline 卡片式；每 task 顯示 4~5 個重點階段 pill。
